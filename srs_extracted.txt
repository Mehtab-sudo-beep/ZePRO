1. Introduction  
1.1 Purpose   
The UG/PG Project Allocation System is an application developed using Flutter and Firebase to 
provide a structured and streamlined workflow for academic project allocation across multiple 
colleges and branches. Administrators manage colleges and departments, and students register 
under their respective college and branch using secure authentication. This SRS document 
outlines the scope and requirements of the system.

1.4 Product Scope  
• 
The UG/PG Project Allocation System is an application developed to automate and manage 
the allocation of academic projects to undergraduate and postgraduate students. 
• 
Its primary purpose is to reduce manual effort, improve efficiency, and ensure a fair and 
transparent project allocation process. 
• 
The system benefits Students, Faculty, Coordinators, and Administrators by streamlining 
project selection, approval, and management activities. 
• 
This software supports institutional academic goals by improving project management 
accountability, and overall academic workflow efficiency.

2. Overall Description  
2.1 Product Perspective  
• 
This product is not part of an existing product family and is being created as a fresh 
solution to address limitations of current manual or informal allocation practices. 
• 
The application is a self-contained system that provides functionality for project proposal, 
allocation management, and vacancy tracking.

2.2 Product Functions  
The application must facilitate the following key functions: 
1. User Registration and Authentication 
Users must be able to securely access the application based on their assigned roles. 
• 
Signup and Login: Users register and log in using authorized credentials. 
• 
Role Based Access: Students, Faculty, Faculty Coordinators, and Administrators have 
different access rights. 
• 
Profile Management: Users can view and update basic personal and academic details. 
• 
Password Recovery: Secure password reset functionality is provided. 
2. Project Creation and Management 
Faculty members manage academic project information through the application. 
• 
Project Proposal: Faculty can create and publish UG/PG project topics. 
• 
Project Details Management: Project title, description, domain, and availability are 
maintained. 
• 
Project Visibility: Project slots are displayed to eligible students based on criteria. 
3. Student Project Selection 
Students interact with the system to choose projects. 
• 
Project Browsing: Students can view available project listings. 
• 
Request Submission: Students can submit and update project requests within a defined 
period. 
• 
Allocation Status: Students can view their assigned project details. 
4. Project Allocation Processing 
The system manages fair and transparent project allocation. 
• 
Allocation Rules: Projects are allocated based on predefined rules and availability of slots 
and meeting outcomes. 
• 
Coordinator Control: Faculty Coordinators oversee and finalize the allocation process.

• 
Conflict Handling: The system supports resolution of allocation conflicts by scheduling 
meeting for all the requests by students. 
5. Supervision 
The application supports academic supervision activities. 
• 
Faculty Supervision: Faculty can view and manage assigned students. 
• 
Faculty Coordinator Supervision: Faculty Coordinator can view remaining non-allocated 
groups/students and faculty who are free. 
• 
Supervision Panel: This panel is visible to Faculty Coordinator to view the status of 
students and faculty. 
6. Administration and Maintenance 
Administrative functions ensure smooth operation of the system. 
• 
User Management: Administrators manage users and roles. 
• 
System Maintenance: Basic configuration and data maintenance features are provided. 
• 
Record Management: The system maintains accurate records of projects and allocations. 
 
2.3 User Classes and Characteristics  
The UG/PG Project Allocation System is intended to be used by different classes of users with 
varying roles, access privileges, and levels of interaction. The major user classes are described 
below: 
• 
Students (UG/PG) 
Students are the primary users of the system. They use the application to view available 
projects, submit project requests, and track allocation status. Students have basic technical 
knowledge and limited access privileges restricted to student related functionalities. 
• 
Faculty Members 
Faculty members use the system to propose project domains and sub-domains, review 
student requests, and lock allocated student slots. They interact with the system and have 
moderate technical expertise with access limited to faculty related features. 
• 
Faculty Coordinators 
Faculty coordinators are key users responsible for overseeing the project allocation process 
and setting the rules for project allocation. This user class has higher privileges and 
frequently interacts with students as they have a separate dashboard for viewing the non-
allocated students and the faculties who are free to allocate students under them.

Administrators 
Administrators manage user accounts, roles, system configuration, and maintenance 
activities. They have the highest level of access and technical expertise but use the system 
less frequently compared to coordinators.

5.Other Nonfunctional Requirements  
5.1 Performance Requirements  
• 
The application should load the dashboard/home screen within 5 seconds under a stable 
4G or Wi-Fi connection. 
• 
Project listings, faculty profiles, and slot availability details should be displayed within 5 
seconds after a user request. 
• 
The system should support simultaneous access by up to 1000 users without significant 
performance degradation. 
• 
Notifications such as project request updates, approvals, rejections, and alerts should 
be delivered within 4-5 seconds of the triggering event. 
• 
Critical operations such as team formation, faculty request submission, approval 
actions, and slot updates should be completed within 6–8 seconds.

Rationale 
Slightly relaxed performance limits ensure stable operation under varying network conditions and 
peak academic usage periods, while still providing a responsive and reliable user experience for 
students, faculty, coordinators, and administrators. 
5.2 Safety Requirements  
• 
The system shall prevent data loss or inconsistency during critical operations such as slot 
updates, faculty approvals, and team allocations. 
• 
The application shall validate all inputs to prevent invalid team formation, exceeding slot 
limits, or rule violations defined by the Faculty Coordinator. 
• 
The system shall ensure that slot availability updates occur atomically, so partial updates do 
not result in incorrect allocation states. 
• 
In case of system failure, network interruption, or unexpected shutdown, the application 
shall ensure that previously saved data remains intact. 
• 
The system shall provide confirmation prompts before performing irreversible actions such 
as request cancellation or record deletion. 
• 
The application shall restrict access to critical operations (deadline overrides, coordinator 
role changes, record removal) to authorized users only. 
• 
The system shall display clear error messages and guidance to prevent unsafe or 
unintended user actions. 
 
5.3 Security Requirements  
• 
The system shall require secure user authentication for all users (Students, Faculty, 
Faculty Coordinators, and Administrators) before granting access to any application 
features. 
• 
The application shall implement role-based access control (RBAC) to ensure users can 
access only the data and functions permitted for their assigned role. 
• 
All communication between the mobile application and backend services shall be secured 
using HTTPS with SSL/TLS encryption to protect data in transit. 
• 
User credentials and authentication tokens shall be securely managed using Firebase 
Authentication, preventing unauthorized access. 
• 
Sensitive user data such as academic documents (marksheets, resumes) shall be 
stored securely with controlled access and shall not be accessible to unauthorized users.

•

5.4 Software Quality Attributes  
• 
Availability: The application shall maintain high availability (around 99.5% uptime), 
excluding planned maintenance. 
• 
Reliability: The system shall recover gracefully from crashes and preserve unsent actions. 
• 
Maintainability: The system shall use a modular design to allow easy updates and bug 
fixes. 
• 
Usability: The application shall be easy to use, enabling users to complete basic tasks with 
minimal effort. 
• 
Scalability: The system shall support growth in users, institutions, and departments using 
a scalable cloud backend. 
• 
Portability: The application shall run on Android devices and can be extended to other 
platforms with minimal changes. 
• 
Testability: The system shall support automated testing to ensure correctness and stability. 
 
5.5 Business Rules  
• 
Only verified users can access the application based on assigned roles. 
• 
Faculty members can supervise students only within allocated slot limits. 
• 
Student teams can send requests only within the defined deadline. 
• 
Slot availability shall update automatically after approval or rejection. 
• 
Approved requests cannot be modified by students. 
• 
Administrators can add, remove, or modify users and roles when required. 
• 
All allocation activities shall be logged for tracking and audit purposes.
